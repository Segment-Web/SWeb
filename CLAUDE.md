# CLAUDE.md — контекст проекта Segment

Этот файл — точка входа для новой сессии. Читается автоматически. Здесь: что за
проект, как устроен, что уже сделано, что дальше и как проверять.

## Что это

**Segment** — вайбкод-мессенджер с открытым кодом. Реалтайм-чат по WebSocket,
панельный интерфейс, сквозное шифрование (E2EE). Монорепо на npm workspaces.

**Главная идея проекта:** расширяемость **модификациями** — интерфейс собран из
самостоятельных панелей, которые можно двигать/ресайзить/менять местами; в
перспективе — библиотека пользовательских модов (часть требует установки у всех
участников чата). Дизайн — спокойная тёмная тема, без пестроты и градиентов.

## Как запускать и проверять

- `npm install` (workspaces), затем `npm start` → http://localhost:3000.
- Точка входа сервера: `apps/server/src/index.js`. Порт 3000 (env `PORT`).
- Превью настроено в `.claude/launch.json` (имя конфигурации `segment`) — для
  preview_* инструментов.
- Крипто-самотест: `node packages/crypto/selftest.js` (сейчас **17/17**).

**Важно про headless-превью:** визуальный вьюпорт нулевой ширины —
`elementFromPoint` не работает и скриншоты таймаутятся. Логику перетаскивания
панелей тестировать через принудительную ширину (`workspace.style.width=...`) и
геометрию (`getBoundingClientRect`), а не через хит-тест/скриншоты. Многоклиентный
E2EE тестировать, создавая несколько `SegmentClient` в одной странице через
`import('/shared/core/index.js')`.

**Состояние:** сервер историю НЕ хранит (E2EE, слепой ретранслятор) — перезапуск
сервера очищает всё. localStorage клиента: `segment_name`, `segment_saved`.

## Архитектура (слои, по мотивам открытого кода Telegram)

```
packages/protocol  — «TL-схема»: типы сообщений, комнаты, лимиты, PROTOCOL_VERSION (=2)
packages/core      — «TDLib»: соединение, состояние, поток обновлений, E2EE-рукопожатие; без DOM
packages/crypto    — Segment Secure Layer: наш E2EE на WebCrypto (см. ниже)
apps/server        — HTTP (статика) + WebSocket = слепой ретранслятор шифртекста
apps/web           — тонкий UI поверх ядра: панели + тайлинг-раскладка
```

- Ядро (`@segment/core`) — событийный API: `client.on(event, cb)`. События:
  `connection · identity · status · chats · room · append · typing`. Команды:
  `connect · setIdentity · openRoom · send · notifyTyping`. Снимок для UI: `view()`.
- Единый исходник пакетов: в Node — npm workspaces; в браузере — **import map** в
  `apps/web/public/index.html` (`@segment/protocol|core|crypto` → `/shared/...`),
  которые сервер раздаёт из `packages/` (`apps/server/src/static.js`).
- Подробности: `ARCHITECTURE.md`, крипта — `docs/encryption.md`.

## Веб-клиент: панели и раскладка

- Панель = `{ id, title, mount(body), weight? }`; регистрируется в реестре
  (`apps/web/public/js/panels/registry.js`) — это зачаток системы модов.
- Встроенные панели: `profile` (аватар, ник, статус, шестерёнка настроек —
  **заглушка, onclick пустой**), `chat-list`, `chat-room` (в шапке — аватар).
- Раскладка `apps/web/public/js/workspace/workspace.js` — **дерево сплитов**
  (не сетка): бросаешь панель на край другой (лево/право/верх/низ) → деление в ту
  сторону; на центр → своп. Ресайз в обе стороны. Плавность — FLIP-анимация.
  Панель профиля не ужимается меньше содержимого (min-width/height по CSS,
  ресайз читает индивидуальный минимум). Есть `window.Segment = {client, registry, workspace}`.

## Шифрование (E2EE) — сделано

Наш слой `packages/crypto` («Segment Secure Layer») — свой протокол на
**стандартных примитивах** (AES-256-GCM, ECDH P-256, HKDF-SHA-256, ECDSA для
подписи prekey). Без сторонних крипто-библиотек.

Готово (план — в `docs/encryption.md`):
1. Примитивы + sender-key для комнат (forward secrecy).
2. Сервер = слепой ретранслятор (в ретрансляции только `iv`+`ct`, поля `text` нет).
3. **Double Ratchet** (двойной храповик) — post-compromise security, обработка
   сообщений вне очереди.
4. **X3DH + каталог prekeys** — установление сессии без онлайна собеседника;
   подпись signed-prekey проверяется.
5. **Ротация sender-key** при выходе участника (ушедший не читает будущее).
6. Гибрид PQXDH — **только задел**: обвязка под пост-квантовый KEM готова и
   протестирована мок-объектом, но настоящего ML-KEM нет (в WebCrypto его нет, а
   руками писать нельзя). По умолчанию — классика.

Протокол v2, типы: `Join(bundle)/Roster/Peer/PeerLeft/PreKeyRequest/PreKey/KeyShare/Cipher/Typing/System`.
Рукопожатие: меньший `id` инициирует X3DH (запрашивает у сервера одноразовый
prekey), открывает храповик и первым отдаёт sender-key; второй восстанавливает
секрет из X3DH-заголовка и отвечает своим. Сообщения в комнаты — sender-key.

> ⚠️ Крипта — **рабочий прототип**, перед продом обязателен независимый аудит.
> Метаданные (кто/когда/куда, «печатает…») сервер видит. Истории нет (новый
> участник видит сообщения только с момента входа).
